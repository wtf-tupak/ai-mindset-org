# POS rule Session Management — 2026-04-18

## Жизненный цикл сессии

```
[SessionStart] → [Работа] → [Контроль контекста] → [Завершение] → [Handoff]
```

---

## 1. Session Start

### При входе в сессию

1. **Проверить hooks** — автоматически запустятся:
   - Linear Awareness (активные задачи)
   - Ambient Research (фоновые исследования)

2. **Восстановить контекст** (если есть):
   ```
   /continue {session-name}
   ```

3. **Запустить morning brief** (если сессия утром):
   ```
   /pos-morning
   ```

---

## 2. Контроль контекста

### Триггеры по уровню контекста

| Символ | Уровень | Действие |
|--------|---------|----------|
| ● OK | <50% | Работаем нормально |
| ◐ WARN | 50-70% | **Сохранить чекпоинт** `/session-save {name}` |
| ○ CRIT | >70% | **Срочный handoff** — сохранить и остановить |

### Футер сессии

Каждый ответ содержит:
```
◇ ctx: 45K | 155K left | 23% | ●
```

Интерпретация:
- `usedK` — сколько использовано
- `leftK` — сколько осталось
- `%` — процент использования
- `● ◐ ○` — статус (OK / WARN / CRIT)

---

## 3. Сохранение сессии

### Когда сохранять

- **◐ WARN** (50-70%) — профилактическое сохранение
- **Завершение задачи** — логическая точка
- **Переключение контекста** — новая задача
- **Планируемый перерыв** — >30 минут

### Как сохранять

```bash
# С именем (рекомендуется)
/session-save {task-name}

# Без имени — авто-генерация
/session-save
```

### Именование сессий

```
{тема}-{MM DD}
```

Примеры:
- `dashboard-gen-0418`
- `linear-sync-0418`
- `pos-audit-0418`

Правила:
- lowercase
- hyphen-separated
- max 40 chars
- тема + дата

---

## 4. Завершение сессии

### Что происходит автоматически

Хук `session-log`:
- Логирует Linear task references
- Сохраняет в `~/.claude/sessions/`

### Ручные действия

1. **Сохранить если не сохраняли**:
   ```
   /session-save {name}
   ```

2. **Проверить dashboard** (опционально):
   ```
   /pos-dashboard-gen
   ```

3. **Подтвердить handoff** — убедиться что чекпоинт доступен:
   ```
   ls ~/.claude/projects/{project}/memory/sessions/
   ```

---

## 5. Handoff Protocol

### Что такое handoff

Передача контекста между сессиями. Состоит из:
- Сохранённый чекпоинт (`/session-save`)
- Компрессированное описание состояния
- Продолжение через `/continue`

### Когда делать handoff

- Контекст >70% (CRIT)
- Задача не завершена, но сессия истекает
- Переключение на другую задачу
- End of day — сохранить текущее состояние

### Процесс handoff

```
1. Оценить: контекст >70%?
2. Сохранить: /session-save {name}
3. Подтвердить: файл создан?
4. Выйти: сессия завершена
5. Новая сессия: /continue {name}
```

---

## 6. Восстановление сессии

### Команды

```bash
# Последняя сессия
/continue

# Именованная сессия
/continue {name}

# Показать доступные
ls ~/.claude/projects/{project}/memory/sessions/
```

### Что восстанавливается

- Цель сессии
- Что было сделано
- Что в pending
- Изменённые файлы
- Ключевые решения
- Контекст для продолжения

---

## 7. Принципы

- **Частые чекпоинты** — лучше сохранить лишний раз
- **Именование** — всегда с именем, легко найти
- **Handoff чистый** — одна сессия = одна задача
- **Контекст не бесконечный** — 75K токенов hard limit
- **Hooks работают** — не отключать без причины

---

## 8. Анти-паттерны

| Не делать | Почему | Вместо этого |
|-----------|--------|--------------|
| Ждать 100% контекста | Потеряем последние данные | Сохранять на 60-70% |
| Длинные сессии >3 часов | Сложно восстановить контекст | Делить на чанки |
| Безымянные сессии | Невозможно найти | Всегда с именем |
| Игнорировать ◐ WARN | Риск потери контекста | Сохранять сразу |
| Работать в >70% | Непредсказуемое поведение | Handoff + новая сессия |

## 9. Дополнительные ограничения

### Лимит ходов

- **Максимум 20-25 ходов** на сессию
- После 25 — `/compact` или новая сессия
- Причина: качество деградирует с длиной контекста

### Параллельная работа

**Subagent'ы запускать параллельно** если:
- 2+ независимые задачи
- Нет зависимостей между ними
- Можно объединить результаты

Пример:
```
[Task A] ─┐
          ├→ [Wait] → [Merge results]
[Task B] ─┘
```

### Фоновые задачи

**Для долгих операций** использовать `run_in_background: true`:
- Транскрипция аудио
- Деплой на сервер
- Длинные сканирования
- Фоновые исследования

Пример:
```json
{
  "tool": "Agent",
  "run_in_background": true,
  "prompt": "..."
}
```

Результат приходит уведомлением, не блокирует сессию.
